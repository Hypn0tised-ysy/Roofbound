# Roofbound 协作说明

本文档用于统一团队在本项目中的开发流程、提交规范与协作约定，减少冲突并提升可维护性。

## 1. 分支与协作流程

1. `main` 分支：仅保留可运行、可发布的稳定版本。
2. 功能开发请从 `main` 拉取个人分支：
	- `feature/功能名`（新功能）
	- `fix/问题名`（缺陷修复）
	- `chore/事项名`（工程配置、脚本整理等）
3. 开发完成后发起 Pull Request（PR）到 `main`，通过评审后再合并。
4. 禁止直接向 `main` 提交代码。
5. 长周期分支请每天同步 `main`，避免集中解决大冲突。

## 2. Git Commit 规范

### 2.1 提交信息格式

推荐使用 Conventional Commits：

`<type>(<scope>): <subject>`

示例：

- `feat(player): 增加二段跳能力`
- `fix(ui): 修复暂停菜单按钮失效`
- `refactor(enemy): 拆分巡逻与追击状态逻辑`
- `chore(project): 调整输入系统配置`
- `docs(readme): 更新启动说明`

### 2.2 type 建议

- `feat`：新功能
- `fix`：修复缺陷
- `refactor`：重构（不改变对外行为）
- `perf`：性能优化
- `test`：测试相关
- `docs`：文档变更
- `chore`：构建、配置、依赖、工程维护
- `style`：不影响逻辑的格式调整

### 2.3 提交粒度

1. 一次 commit 只做一件事（单一目的）。
2. 提交前确保工程可打开、场景可运行。
3. 不要把无关改动（自动格式化、临时调试代码）混在同一个 commit。

## 3. Pull Request 规范

PR 描述建议包含：

1. 变更目的：本次要解决的问题。
2. 主要改动：脚本、场景、预制体、配置等变化点。
3. 验证方式：如何复现、如何验证修复或功能。
4. 风险说明：可能影响的系统与回滚方式。

建议标题格式：

- `feat: 玩家冲刺机制`
- `fix: 关卡2出生点错误`

## 4. Unity 项目文件规范

### 4.1 必须提交

1. `Assets/`
2. `Packages/`
3. `ProjectSettings/`
4. 所有 `.meta` 文件

### 4.2 不应提交

1. `Library/`
2. `Temp/`
3. `Logs/`
4. `UserSettings/`

说明：以上目录通常应由 `.gitignore` 忽略，避免仓库膨胀与环境污染。

### 4.3 资源与命名

1. 场景放在 `Assets/Scenes/`，命名清晰，如 `MainMenu.unity`、`Level01.unity`。
2. 预制体使用统一前缀，如 `PF_Player`、`PF_EnemySlime`。
3. 脚本文件名与类名必须一致。
4. 避免重名资源，防止合并时引用混乱。

## 5. 开发规范

1. 禁止提交调试垃圾：`Debug.Log` 风暴、临时代码、废弃脚本。
2. 公共系统改动需补充注释和使用说明。
3. 复杂逻辑优先拆分为小模块，避免超长类和超长方法。
4. 修改输入、层级、Tag、物理参数时，在 PR 中明确说明影响范围。
5. 场景和 Prefab 变更冲突概率高，提交前先同步并本地验证。

## 6. 合并冲突处理建议

1. 优先在本地解决冲突并完整运行项目后再推送。
2. Scene、Prefab 冲突若无法确认，及时与相关同学结对处理。
3. 不确定的冲突不要“凭感觉”保留，必须在 Unity 中打开检查引用是否正确。

## 7. 发布前自检清单

1. 工程可正常打开，无明显报错。
2. 本次改动功能可复现、可验证。
3. 无无关文件和临时文件被提交。
4. commit 信息符合规范。
5. PR 描述完整，便于评审与回归测试。
